# Architecture Artifacts

This directory contains the architecture source of truth for the repository.

## Source files

These files are edited directly:

- `ownership-matrix.md`
- `lifecycle-map.md`
- `recovery-map.md`
- `target-architecture.template.md`
- `elements/*.md`
- `context-settings.json`

Each file under `elements/` is one architectural element and is the authoritative place to define:

- responsibility
- collaborators
- code ownership hints
- invariants
- lifecycle notes
- branch aliases for focus selection

## Generated files

These files are generated by the repository workflows and should not be edited manually:

- `target-architecture.md`
- `contexts/*.xml`
- `context-manifests/*.md`

Generated artifacts are refreshed on non-main branches and committed back automatically.

## Focus element workflow

Each meaningful implementation branch should map to one architectural element through one of that element’s `branch_aliases`.

The focus element is used to:

- select the current branch’s focused architectural context
- generate `contexts/current-focus.context.xml`
- determine which architecture file must be updated before code changes land

If a branch name does not match any alias, the repository falls back to the default focus element configured in `context-settings.json`.

## Branch naming guidance

Use branch names that clearly include one of the configured aliases, for example:

- `feat/library-service-folder-connection`
- `feat/player-session-service-bottom-drawer`
- `chore/architecture-governance-context-automation`

## Generation flow

GitHub Actions performs the following automatically:

1. validate architectural metadata and collaborator references
2. infer the focus element from the branch name
3. regenerate `target-architecture.md`
4. regenerate repository and per-element code-path coverage manifests
5. regenerate per-element focused context XML files
6. regenerate `contexts/current-focus.context.xml`
7. regenerate `repo-context.xml` and `ci-context.xml`
8. commit refreshed generated artifacts back to non-main branches

## Context artifacts

The repository produces three kinds of generated context artifacts:

- `repo-context.xml` for broad repository context
- `contexts/*.xml` for focused architectural slices
- `context-manifests/*.md` for code-path coverage and filesystem visibility

The coverage manifests exist to clarify whether architecturally declared code paths are:

- present
- empty
- missing
- unmatched by a declared glob

Use them to distinguish “not included in this packed context” from “not implemented in the repository”.

## Context policy

- `repo-context.xml` is the broad repository context
- `contexts/*.xml` are focused architectural slices
- `context-manifests/*.md` explain architectural code-path coverage
- `repo-context.xml` should include the repository-level coverage manifest
- focused element contexts should include the relevant element coverage manifest
- `repo-context.xml` must not include `contexts/*.xml`
- `repo-context.xml` should not include per-element coverage manifests
- `ownership-matrix.md` should remain small enough to include in all coding sessions
